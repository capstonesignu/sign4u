## 二、#2 敬语(已交付)
- **问题**:训练/测试数据已敬语化,但代码 pipeline 还在强制平语(`~다`),互相冲突。
- **解决**:整合敬语数据 + Layer4/5 补丁(평어→합쇼체)+ 重训 EXAONE-3.5-2.4B(QLoRA 4bit)。
- **结果**:96 句测试 → **敬语率 97.9%**(94/96)、BLEU 0.285、chrF++ 0.475、keyword recall 91%、单句推理 0.36s。
- **产物**:`exaone-finetuned-v4`。

## 三、#3 更大模型(结论:2.4B 更优)
- 按要求测了 EXAONE-3.5-**7.8B**:4bit 推理 0.28s(**<1s 达标 ✅**)。
- 但同样数据微调后,**7.8B 质量反而更差**:

| 模型 | 敬语率 | BLEU | 幻觉率 |
|---|---|---|---|
| **2.4B** | **97.9%** | **0.285** | **56.7%** |
| 7.8B | 91.7% | 0.183 | 64.9% |

- **原因**:7.8B 太"自由发挥"——爱加输入没有的内容、擅自改时态;对"忠实 + 最小转换 + 固定敬语"这种受限任务反而是负担。
- **完整 2×2 对比(敬语 + 候选两个任务都实测,全 4bit)**:

| | 纯敬语(BLEU) | 敬语+候选(候选还原率) |
|---|---|---|
| **2.4B** | **0.285** | **79.8%** |
| 7.8B | 0.183 | 77.2% |

- **结论**:**交付用 2.4B** —— 4 个格子 2.4B 全胜;7.8B 在两个任务上都因"爱加戏/忠实度低"而更差。这是有数据支撑的决策,不是盲目堆参数。
- (注:候选机制对两个尺寸都有效 —— candidate 均 > noisy:2.4B 79.8>65.3、7.8B 77.2>60.5。)

## 三-B、4bit vs 16bit 精度(实测:4bit 反而更好)
同一个 2.4B v4 模型,只改加载精度,实测:

| 指标 | 4bit | 16bit(原始) |
|---|---|---|
| BLEU | **0.283** | 0.217 |
| chrF++ | **0.473** | 0.409 |
| 幻觉率 | **54.4%**(越低越好) | 67.9% |
| 敬语率 | **100%** | 99% |

- **4bit 几乎每项都更好或持平**(96 句里 71 句输出不同)。
- **原因**:模型是 QLoRA —— 适配器在 **4bit 底座**上训练,天生匹配 4bit;放到 16bit 底座反而"水土不服"(训练/部署不匹配)。
- **结论**:用 4bit **不是妥协,而是对本模型最优**(省显存 1/4 + 更快 <1s + 质量更好)→ 完全满足组长"4bit 量化"要求,且有数据支撑。
- ⚠️ 边界:这不代表"4bit 普遍 > 16bit",而是"**QLoRA 训出来的适配器就该在它训练时的 4bit 上跑**"。

## 四、#1 候选生成(验证可行)
- **不枚举**(每位置 3 候选 × N 词 = 3ᴺ 组合爆炸),改用"**候选喂给模型、模型自己挑**"(语言模型重排,机制 B)。
- **关键发现**:模型**必须专门学"候选格式"**,否则会被干扰词带偏。

| 输入方式 | 还原正确词(true-recall) |
|---|---|
| 给正确词(理想上界) | 89.0% |
| 只给一个猜测(noisy baseline) | 65.3% |
| 给候选 —— 朴素(没学候选格式) | 25.1% ❌ |
| **给候选 —— 候选格式微调后** | **79.8%** ✅ |

- **解读**:候选微调后,"给候选"**明显优于"只给一个猜测"**,接近理想上界 → **组长的候选思路可行,关键在候选格式微调**。
- **产物**:`exaone-finetuned-v4-cand`(50% 干净 + 50% 候选数据训练)。

## 五、架构 & 接口建议
- **保持两阶段**(SLR + sLLM);sLLM 定位为"**鲁棒纠错层**"——用语言上下文修识别错误(类比语音识别的语言模型纠错)。
- **候选数**:每位置 **最多 3 个**,**按把握度排序 + 阈值**(确定的位置 1 个,模糊的才 2–3 个)。
- **需 SLR 团队配合**:输出 top-3 + 把握度 → 详见《SLR-sLLM 接口约定》。
